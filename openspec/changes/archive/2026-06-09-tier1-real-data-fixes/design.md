## 上下文

`bootstrap-parse-skeleton` 的 tier1 正则在真实山姆 HAR(440 商品)上数量召回仅 20.7%(精度 95.8%)。失败集分析出两类纯解析可补的模式:单件大规格(无 `*N`,数量隐含 1)与数量前置(`24x500mL`)。本变更只提召回,改进幅度交 `add-eval-harness` 度量。

## 目标 / 非目标

**目标:**
- 单件容量大规格 → 数量 1,可算 per100ml。
- 数量前置(count-before-size)可正确提取,同时保住上次 `X20` 修复对品名噪声的免疫。
- 不改既有干净标题行为;数量精度(对 `samPkgNum`)不下降。

**非目标:**
- per-100g(重量计价饮料)——需新可比单位,独立 change。
- 改 tier2/tier3/HTTP/置信。纯品名无规格仍交 tier2。

## 决策

### D1:单件推断只在「有容量 + 零数量信号」时触发
**选择**:`unitSize(ml/L)` 在场,且无乘号(`*×x`)、无包装计数(`N瓶/罐/支/盒/袋`)→ `quantity=1`、`totalAmount=unitSize→ml`。
**理由**:山姆单件大规格(`4L`、`4.104升`)普遍;数量 1 是确定且安全的推断(不臆造多件)。
**风险**:误把「应有数量但没抽到」当单件 → 高估 per100ml。缓解:仅当**完全无任何数量信号**时触发;若有乘号/包装计数但没解析出,走原缺项路径而非强行置 1。

### D2:count-before-size 用「紧贴 size 的前缀」识别,不放开全标题搜索
**选择**:在 SIZE_RE 命中后,数量优先在 size **之后**子串搜(现行);若后侧无,再在 size **紧邻之前**的窗口里找 `(\d+)\s*[*×x]\s*$` 形态(如 `24x` 紧贴 `500mL`)。
**理由**:既支持 `24x500mL`,又不退回「全标题搜」——后者正是 `X20` bug 的根源。把前置识别限制在「紧贴 size」可两全。
**替代**:全标题双向搜 —— 否决,重新引入品名噪声误判。
**诚实边界**:`整箱24听 可乐 330ml`(数量与 size 隔着品名)这类仍不覆盖——交 tier2;本次只收「数量紧贴 size」的前置形态。

## 风险 / 权衡

- **单件推断高估风险**(D1):靠「零数量信号」严格门控 + eval 基线回归(数量精度不得跌)兜底。
- **前置正则与既有后置逻辑交互**:须保证 `330ml*24听`(后置)、`24x500mL`(前置)、`可口可乐X20 330ml*6听`(噪声)三者并存正确——全部进 parser 单测。
- **真实标题长尾**:本次只覆盖两类高频模式,其余继续由 tier2 + confidence 兜底,不追求 tier1 全覆盖。

## 未决问题

- 「紧贴 size 之前」的窗口宽度/正则细节(是否允许中间一个空格、是否允许 `提/箱` 量词)——实现时用真实样本调,以 eval 数量精度为准绳。
