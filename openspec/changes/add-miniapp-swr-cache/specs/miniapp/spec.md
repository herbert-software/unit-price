## 新增需求

### 需求:榜单首屏必须经端上 SWR 缓存即时回显并后台重验

榜单首屏**必须**支持端上 stale-while-revalidate:榜单页**新挂载**(冷启动 / 重开小程序 / `navigateTo` 进入,无保留内存态)且该 cohort 有有效端上缓存时,**必须**先即时渲染缓存快照、再后台重验,使用户在边缘冷尾(CDN purge 后 / TTL 过期 / POP 驱逐)与弱网下也能秒显上次榜单而非白屏等待。具体**必须**满足:

- **作用时机**:仅钩冷启动/新挂载的首屏加载(常驻 tabBar 页 `useLoad→loadFirst`@`idle` 一生一次);切 Tab/前后台回访由保留内存态即时呈现,**不**在本需求(自动 re-validate-on-re-entry 是非目标)。
- **缓存范围与键**:仅缓存榜单首页(`offset=0`、`q===undefined`——`useRankings` 边界已解析/截断后的值,**非**原始 query key 存在性)的已校验 `RankingsItem[]` 快照;键 = **客户端自身 `category` 入参**,缺省(`category===undefined`,落地榜)归一到一个**固定哨兵字面量**(客户端不镜像、也拿不到服务端 `'soft-drink'` 默认)。搜索(`q` 校验后有效)与翻页(`offset>0`)**禁止**缓存(与服务端对有效 `q` 发 `no-store` 同口径)。
- **读即重校验**:读缓存**必须**经 `parseRankingsResponse(raw)` 重新校验(**单参**——`jitless` 已在该函数内部写死、`raw` 在 weapp 安全,**禁止**传第二参);损坏 / 旧 schema 残留 → **必须**视作未命中,**禁止**渲染未校验脏数据。
- **命中即时回显**:命中有效缓存时首屏**必须**跳过整屏 loading 态、直接以 `ready` 渲染旧快照,并**必须**同步把分页游标置为 `cached.length`(否则随后触底 `runNext` 会从 `offset=0` 重取、首页重复追加);随后**必须**后台重取首页;重验成功**必须**覆盖列表、游标与该 cohort 缓存。
- **重验失败保留旧数据**:命中缓存后台重验**失败**时**必须**保留旧快照、停留 `ready`,**禁止**因首屏错而清屏(实现**必须**改 `runFirst` 的 catch 按 live `items` 分叉、镜像既有 `refresh` catch 的"有列表则保 ready、否则整屏错"范式,而非当前无条件清空)。
- **缓存缺失不变**:无有效缓存(首次安装 / 已清 / 损坏)时,首屏行为**必须**与既有"榜单一屏必须支持分页与加载/空/错三态"需求完全一致(loading → ready/空,首屏失败 → 整屏错误态 + 重试),本需求**不**改其行为。

#### 场景:命中缓存即时回显 + 后台刷新覆盖

- **当** 冷启动 / 重开小程序、榜单页新挂载且该 cohort 存在有效端上缓存
- **那么** **必须**立即以缓存快照 `ready` 渲染(无整屏 loading 态)、并同步置分页游标为 `cached.length`,再后台重取首页;fresh 到达后**必须**覆盖列表、游标与该 cohort 缓存

#### 场景:命中后触底翻页不重复首页

- **当** 命中缓存即时回显后(后台重验未回或失败),用户触底翻页
- **那么** `runNext` **必须**从 `offset=cached.length` 续取下一页、**禁止**从 `offset=0` 重取导致首页行重复追加

#### 场景:有缓存时后台重验失败保留旧数据

- **当** 命中缓存后,后台首页 fetch 失败或 `parseRankingsResponse` 校验失败
- **那么** **必须**保留缓存快照、停留 `ready`,**禁止**转整屏首屏错或白屏

#### 场景:缓存缺失回落既有三态

- **当** 无有效缓存(首次安装 / 已清 / 缓存损坏)进入榜单
- **那么** 首屏**必须**按既有三态:loading → ready/空,首屏加载或校验失败 → 整屏错误态 + 重试入口,行为不变

#### 场景:仅缓存非搜索首页

- **当** 用户发起搜索(`q` 校验后有效)或触底翻页(`offset>0`)
- **那么** **禁止**读写端上缓存;仅 cohort 榜单首页(`offset=0`、无有效 `q`)的快照可入缓存
