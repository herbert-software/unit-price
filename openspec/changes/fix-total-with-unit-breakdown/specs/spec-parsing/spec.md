## 新增需求

### 需求:总量复述（前导总量 + 单件×数量）须经自洽校验后才改绑 unitSize

当标题里数量乘号 `[*×x]<N>`(本判定只取**第一个 size 之后**的首个乘号——即既有 `QTY_RE` 在 `title.slice(sizeEnd)` 的首个命中;`N>0`)的**左侧窗口**(从标题起点到该乘号)出现**两个或更多** size token 时:窗口内**最靠后的** size 是「单件容量」候选(`单件Vol`),窗口内**第一个** size 是「前导总量」候选(`前导Vol`)(二者之间夹的其它 size 一律忽略)。**仅当**(a) `单件Vol` 与 `前导Vol` **都为体积单位**、(b) `前导Vol` 换算到 ml 后 **> 0**(除零守卫)、(c) `N>0`、(d)「单件Vol × N」与「前导Vol」**换算到 ml 后在容差内相等**(自洽)——四者**全部满足**时,tier1 **必须**判定为「总量复述」格式(典型 `2.1L(100mL×21)`、`2L装可乐 330ml*6`),按括号拆解改绑:`unitSize = 单件Vol`、`quantity = N`、`totalAmount = 单件Vol × N`(沿用 `单件Vol` 单位、parsing 阶段不跨 `ml`↔`L` 换算)。**禁止**把前导总量当 `unitSize` 再乘 `N`(那会把已是总量的值再乘一遍、使单价偏差 `N` 倍)。

**自洽校验是触发门、不是仅 warning**:当「单件Vol × N」与「前导Vol」**不自洽**(换算后相对误差超容差,如 `550mL便携装 1.5L*6` 中 `1.5L×6=9000ml` 与前导 `550ml` 严重不符)时,前导 size **极可能是品名营销词**(`2L装`/`便携550mL`/`买2L送`)而**非**总量复述,tier1 **必须不改绑**——保持「取第一个 size 为 `unitSize`、其后乘号为 `quantity`」的既有行为(不引入回归)。**禁止**在不自洽时仍采纳括号拆解。

**容差**:相对误差 `|单件Vol×N − 前导Vol_ml| / 前导Vol_ml ≤ 0.1`(±10%,含等号,容纳 `2L装`=2000 对 `330ml×6`=1980 这类约 1% 的标签取整差,同时拒绝品名噪声的数量级偏差)视为自洽。

**除零 / 非正 / 非体积守卫(防 NaN/Infinity)**:`前导Vol` 换算到 ml `≤ 0`(如 `0L`,`toMl=0`)、或任一候选为非体积(`toMl` 返回 null,如重量 `g/kg`)、或 `N≤0` 时,**一律判定门不通过、不改绑**——**禁止**对 `前导Vol_ml=0` 做除法(否则产生 `Infinity`/`NaN`,违反 core「绝不输出 NaN/Infinity」硬不变量)。即上面 (a)(b)(c) 任一不满足时,在比较 (d) **之前**短路为「不改绑」。重量等非体积标题不改绑,其单价终态本就 `per100ml=null`,不产生错误单价。

**已知残留(非目标)**:(1) `前导Vol` 是**真总量但标签印刷误差超 10%**(如 `2L(330ml×5)`,1650 与 2000 差 17.5%)时,门挡住、不改绑、回到既有「`unitSize=前导`」行为——此类**仍误算**(总量×N),本变更**不修**(无法区分「印刷错的真总量」与「品名噪声」,自洽门保守地都不改绑)。(2) 前导是**体积品名营销词但数值碰巧与 `单件×N` 自洽**(如 `2L装` 对 `330ml×6`=1980≈2000)时会被当总量改绑——此例结果**恰好正确**,但属「自洽即采纳」的启发式巧合、非真总量识别。**对称性说明**:重量品名词(`2kg礼盒`)因非体积被 (a) 直接挡而保守不改绑(静默 null),体积品名词(`2L装`)因可换算且数值自洽而被采纳——差异仅在「前导是否体积 + 是否自洽」,规则一致(都只在「体积+自洽」时改绑)。本变更只确保**自洽**的总量复述被正确解析、且**不自洽/非体积**的品名噪声不被误改绑;不声称修复一切「总量(单件×N)」。

**与既有规则不冲突**:只有一个 volume size token 时(如 `300mL*24`、`330ml*24听`——乘号左窗口只此一个 size),不进入本判定、行为**不变**;数量前置 `24x500mL`(乘号在 size 之前、size 后无乘号,`24` 无单位不是 size)按既有「数量前置」规则不变;`24x500mL*6` 仍取后侧 `quantity=6`;品名噪声 `可口可乐X20 330ml*6听`(`X20` 不是 size token)仍取紧贴 size 的 `*6`。本判定确定性进行、不调用 LLM,且**只在自洽校验通过**时改变绑定。

#### 场景:总量括号复述自洽——括号内单件×数量为准

- **当** 输入标题为 `多维刺梨柠檬饮 2.1L(100mL×21)`、价格为 `69.9`(`100mL×21=2100ml` 与前导 `2.1L=2100ml` 自洽)
- **那么** tier1 **必须**解析出 `unitSize = {value: 100, unit: "ml"}`、`quantity = 21`、`totalAmount = {value: 2100, unit: "ml"}`,据此算得 `per100ml ≈ 3.33`;**禁止**得出 `2.1L/44.1L/per100ml≈0.159`

#### 场景:取整标签在容差内仍判自洽

- **当** 输入标题为 `2L装可乐 330ml*6`(前导 `2L=2000ml`,`330ml×6=1980ml`,相对误差 1% ≤ 10%)
- **那么** 判自洽,改绑 `unitSize=330ml`、`quantity=6`、`totalAmount=1980ml`(而非把品名 `2L装` 当 unitSize 算成 12L)

#### 场景:不自洽（品名 size 噪声）不改绑

- **当** 输入标题为 `550mL便携装 1.5L*6`(前导 `550ml`,`1.5L×6=9000ml`,严重不符)
- **那么** tier1 **必须不**改绑——保持既有「取第一个 size `550mL` 为 unitSize、`*6` 为 quantity」的行为(`550mL` 是真单件、`1.5L` 视为噪声不参与);**禁止**误把 `1.5L` 当单件、`550mL` 当总量复述

#### 场景:非体积前导不改绑

- **当** 输入标题为 `某蛋白粉 2kg(100g×20)`(前导/单件均为重量 `g/kg`)
- **那么** tier1 **不**进行总量复述改绑(`toMl` 对重量返回 null、无法自洽校验);该商品按既有重量路径走、单价终态 `per100ml=null`(不改绑不产生错误单价)

#### 场景:前导重量 + 真体积单件,保守不改绑(静默 null)

- **当** 输入标题为 `2kg礼盒 330ml*6`(前导 `2kg` 为重量噪声,真单件是体积 `330ml`)
- **那么** tier1 **不**改绑(前导 `2kg` 非体积、自洽门第 (a) 条不满足),保持既有「`unitSize=2kg`(首个 size)、`quantity=6`」;终态 `per100ml=null`(重量不可算)。这是**保守取舍**:本变更不把前导重量当噪声而改绑到体积单件(避免引入「前导非体积时也猜单件」的新启发式),代价是这条可算饮料被保守地落为不可算——属已知、可接受的非目标

#### 场景:前导总量为零,除零守卫不改绑

- **当** 输入标题为 `0L(100mL×21)`(前导 `0L`,`toMl=0`)
- **那么** tier1 **必须不**改绑(除零守卫:`前导Vol_ml ≤ 0` 在比较前短路),**禁止**对 `前导Vol_ml=0` 求相对误差(不得产生 `Infinity`/`NaN`)

#### 场景:普通单件乘数量格式行为不变

- **当** 输入标题为 `可口可乐 汽水小支300mL*24`(乘号左窗口只一个 size token)
- **那么** tier1 仍解析出 `unitSize=300ml`、`quantity=24`、`totalAmount=7200ml`(不进入本判定,行为与引入前一致)

#### 场景:数量前置不受影响

- **当** 输入标题为 `阿尔卑斯山气泡水 24x500mL`(乘号在 size 之前、size 后无乘号)
- **那么** 仍按既有「数量前置」规则得 `unitSize=500ml`、`quantity=24`、`totalAmount=12000ml`

#### 场景:多乘号只取首个及其左最近 size

- **当** 标题含多个乘号(如 `2.1L(100mL×21)` 之后又跟一个乘号),tier1 取**首个**乘号 `×21` 及其左窗口最靠后的 size
- **那么** 其余乘号/size 忽略,得确定结果(`unitSize=100ml`/`quantity=21`);本次不处理「多组单件×数量」的复杂组合(留后续)
