# eval-harness

## 目的

提供单价解析/计算链路的离线评测能力：定义黄金集语料格式、从抓包 HAR 提取语料、对语料逐条跑 `@unit-price/core` 打分并输出机器/人可读指标、以及与基线对比的回归门。本节为待定占位，详见各需求。

## 需求

### 需求:黄金集语料格式

系统必须定义由 Zod schema 描述的黄金集语料格式(JSONL,每行一个样本),TypeScript 类型从 schema 推导。每个样本必须含 `title`(非空字符串)与 `source`(来源标记,如 `har:<file>` / `manual`);可选含 `priceCents`(整数,分)、真值字段(`samPkgNum?` 整数、`samPkgUnit?` 字符串、`samUnitPrice?` **数值**(每件价,元)、`isCompare?` 布尔)、以及人工标注的 `expected`(期望规格,部分字段可空)。`samUnitPrice` 必须是已解析的**数值**(非展示串)——由提取器从山姆展示串 `smallPackagePriceDisplay` 解析得出(见 HAR 提取器)。`isCompare?` 为山姆自带的可比标志,本次仅作为**真值占位字段**收录(无指标消费),留待后续「可比性判断」能力的评测使用。原始山姆全量数据禁止入库;语料文件与原始 HAR 必须被 gitignore。

#### 场景:加载并校验语料

- **当** 读取一个 JSONL 语料文件,其中某行缺 `title` 或 `title` 为空
- **那么** 加载器必须以该行的明确错误拒绝(指出行号),禁止静默跳过或当作有效样本

#### 场景:真值字段可选

- **当** 某样本只有 `title` 与 `priceCents`、无任何 `sam*` 真值或 `expected`
- **那么** 该样本必须仍可被跑批器接受并计入「召回」类指标(只是不计入需要真值的「精度/误差」类指标)

### 需求:HAR 提取器

系统必须提供从抓包 HAR 生成语料的提取器:解析 HAR,从山姆商品列表响应(如 `api-sams.walmartmobile.cn` 的 `goods-portal/grouping/list`)按 `spuId` 去重提取商品,**仅取校准必需字段**(`title`、`priceInfo[].price`、`smallPackageNum`、`smallPackageUnit`、`smallPackagePriceDisplay`、`isCompare`),输出为黄金集语料格式。`smallPackagePriceDisplay` 为展示串(如 `￥18.32/瓶`),提取器必须将其**解析为数值** `samUnitPrice`(每件价,元);当展示串无法解析为单一数值(促销文案/区间价/缺失/非常规格式)时,`samUnitPrice` 置空、该商品计为「无单价真值样本」(仍入语料,只是不计 per-unit 误差),与 `priceCents` 的非数值置空处理对称。提取器禁止把整条原始响应或无关个人/鉴权字段写入语料。`priceInfo[].price` 在山姆响应中为**以分为单位的数值字符串**(如 `"10990"` = ¥109.90),提取器必须按「分」解析为整数 `priceCents`;当 `priceInfo` 为空、缺失或 `price` 非数值时,该商品 `priceCents` 置空、计为「无价格样本」(仍可入语料,只是不计需价格的指标)。同一 `spuId` 多次出现时,必须保留**首次出现**的条目。

#### 场景:从 HAR 提取去重商品

- **当** HAR 含多条 `goods-portal/grouping/list` 响应、商品有重复 `spuId`(字段可能不同)
- **那么** 提取器必须按 `spuId` 去重、保留首次出现条目,每个商品输出一行,带 `source = "har:<filename>"`

#### 场景:跳过非商品响应

- **当** HAR 含与商品无关的响应(配置/埋点/图片等)
- **那么** 提取器必须只提取商品列表响应,忽略其余,不报错

#### 场景:损坏或非 JSON 响应体跳过

- **当** 某条商品列表响应的 body 为截断/非 JSON/无法解析
- **那么** 提取器必须跳过该响应并计数(可在报告中标注跳过数),禁止崩溃或中断整次提取

### 需求:打分跑批器

系统必须提供跑批器:对语料逐条调用 `@unit-price/core` 的 tier1 解析与 tier3 计算,在配置了 `OPENROUTER_API_KEY` 时另跑 tier1+tier2 一档。必须计算并输出以下指标——`unitSize`/`quantity`/`totalAmount` 召回率;**可算率**(tier3 产出非空 `per100ml` 的样本占比,直接度量「多少真实标题能被算出单价」、消费 tier3 计算结论);`quantity` 相对 `samPkgNum` 的精度(仅对有真值样本);**per-unit 价误差**(仅对有 `samUnitPrice`+`quantity` 的样本);tier1-only 与 tier1+tier2 的对比。

**per-unit 价误差的定义**(消除量纲歧义):`@unit-price/core` 只产出 `per100ml`、不产 per-瓶价,而山姆 `smallPackagePriceDisplay` 是 per-瓶/罐价。故跑批器必须用解析出的 `quantity` **自算** `evalPerUnit = (priceCents/100) / quantity`(每瓶价),再与 `samUnitPrice`(数值)比较得误差;**仅当 `quantity` 解出且 `> 0`、且有 `samUnitPrice`(及 `priceCents`)时计入**——`quantity ≤ 0`(如 tier1 对 `330ml*0` 产出的 `0`)与 `null` 同等排除,避免除零。**`per100ml` 无外部真值**:禁止对 `per100ml` 做精度断言,只能由「`quantity` 对 + 容量解析对」间接佐证、并用 per-unit 价误差交叉校验,报告中须如实标注 per100ml 无外部真值。

**禁止 NaN/除零(全局不变式)**:跑批器在**任何**输入下都不得产生 `NaN`/`Infinity`/除零——单样本层面靠上述「`quantity > 0` 才计入 per-unit」保证,指标层面靠空分母处理保证。**空分母处理**:任一指标的分母为 0(空语料、或该指标的合格样本数=0)时,该指标必须记为 `n/a`(非 `0`、非 `NaN`),报告标注「样本不足」,且 `n/a` 指标不参与回归对比。跑批器禁止在无 key 时因缺 key 失败——无 key 时只跑 tier1-only 并在报告中标注 tier2 未评。

#### 场景:无 key 时仅评 tier1

- **当** 运行跑批且 `OPENROUTER_API_KEY` 未配置
- **那么** 跑批器必须正常完成 tier1-only 评测并输出指标,报告中明确标注「tier2 未评(无 key)」,不报错退出

#### 场景:数量精度对照真值

- **当** 某样本 `samPkgNum = 24` 而 tier1 解析出 `quantity = 24`
- **那么** 该样本必须计入 quantity 精度的「正确」;当 tier1 解析出 `quantity = 20`、`0` 或 `null` 时必须计入「错误/缺失」并可列出该样本供排查

#### 场景:per-unit 价误差自算并交叉校验

- **当** 某样本 `priceCents = 10990`、tier1 解析出 `quantity = 6`、`samUnitPrice = 18.32`(由展示串 `￥18.32/瓶` 解析得)
- **那么** 跑批器必须自算 `evalPerUnit = 109.90 / 6 ≈ 18.32` 与 `samUnitPrice` 比较得误差;且禁止对 `per100ml` 报告外部真值精度(仅交叉校验/间接佐证)

#### 场景:quantity 非正样本不计 per-unit 误差

- **当** 某样本 tier1 产出 `quantity = 0`(如 `330ml*0`)且带 `samUnitPrice`
- **那么** 跑批器必须**不计入** per-unit 误差(`quantity ≤ 0` 与 `null` 同等排除),禁止 `priceCents/0` 除零;该样本的 quantity 精度计入「错误」

> 注:`samUnitPrice` 取自山姆展示值 `smallPackagePriceDisplay`,该值含山姆自身的小数舍入(如 `109.90/6=18.3166…` 显示为 `18.32`),由此引入约 0.1% 量级的基底误差。per-unit 误差为相对、阈值可配,基线本身也含同一基底,故不会把正确实现误判回退。

#### 场景:空语料或零合格样本时指标 n/a

- **当** 语料为空、或某指标(如 quantity 精度)的合格样本数为 0
- **那么** 该指标必须记为 `n/a`、报告标注「样本不足」,跑批不得产生 `NaN`/除零错误,该 `n/a` 指标不参与回归对比

#### 场景:报告同时给机器可读与人可读

- **当** 跑批完成
- **那么** 必须输出一份机器可读 metrics(供回归对比)与一份人可读摘要(总召回/精度/误差 + 失败样本清单)

### 需求:回归基线对比

系统必须支持把一次 metrics 快照保存为基线,并在后续跑批时与基线对比:当关键指标相对基线**显著回退**(超过配置阈值)时,以非零退出码 / 明确标记报告回退,供 CI 或本地回归门使用。**回退方向按指标性质分类**(指标有正反向):召回率、quantity 精度等「越高越好」指标,以「低于基线 − 阈值」判回退;per-unit 价误差等「越小越好」指标,以「高于基线 + 阈值」判回退。**首跑无基线**(`baseline.json` 不存在)时,跑批必须正常输出指标、以 `0` 退出、报告标注「无基线,未做回归对比」,不得视为回退。记为 `n/a` 的指标不参与对比。基线快照本身可入库(它是派生指标、非原始数据)。

#### 场景:正向指标回退被拦截

- **当** 改动后 quantity 精度从基线的 95% 跌到 80%(超过回退阈值)
- **那么** 跑批必须以非零退出码 / 明确「回退」标记结束,并列出新增失败的样本

#### 场景:反向指标(误差)升高判为回退

- **当** 改动后 per-unit 价误差从基线的 2% 升到 8%(超过阈值)
- **那么** 必须判为回退(「越小越好」指标按「高于基线 + 阈值」),以非零退出码结束

#### 场景:首跑无基线不算回退

- **当** 运行 `score` 时 `baseline.json` 不存在
- **那么** 跑批必须正常输出指标、以 `0` 退出、报告标注「无基线,未做回归对比」,不得报成回退

#### 场景:准确率提升刷新基线

- **当** 改动后各指标均不低于基线且显式要求更新
- **那么** 系统必须允许把当前 metrics 写为新基线(显式动作,不自动覆盖)
