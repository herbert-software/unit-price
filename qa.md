# QA

Q:
我想开发一个更细的比价系统，先期可以只在山姆会员店类似的商店内比价。这个比价系统相较较常见的比价系统来说，别的比价系统可能会关注同一件商品ID在不同商店的价格，而它要更关注单价。比如说一箱可乐40元，这个比价系统要能理解这一箱可乐有多少瓶、每瓶有多少毫升，然后折算每100毫升大概是多少钱。比如山姆会员店可能会有若干种规格的饮料，它会标注大概一瓶是3元还是4元，但并不关心3元的容量有多大，4元的容量有多大，并且部分商品也不参与比价，这就是我这个比价系统要做的事。我希望这个系统可以做成一个插件式的比价工具，通过接入AI来智能分辨商品规格、和不同口径下的实际单价（一瓶？100mL？）

A:
## 结论先说

这个系统很有价值，而且和传统比价系统不是一个赛道。

传统比价系统核心是：

> **同一个商品 SKU，在 A 店多少钱，在 B 店多少钱。**

你想做的是：

> **不同包装、不同容量、不同组合、不同促销口径下，真实可比较的单位成本是多少。**

这更像是一个：

## **Unit Price Intelligence / 规格归一化比价系统**

中文可以叫：

- **真实单价比价系统**
- **规格归一化比价系统**
- **单位成本比价引擎**
- **家庭消费品性价比分析器**

它特别适合山姆、Costco、盒马、麦德龙、开市客、京东自营、天猫超市这种“规格大、组合复杂、促销口径多”的场景。

---

# 一、它和普通比价系统的本质区别

普通比价系统关心的是：

| 维度 | 普通比价系统 | 你的系统 |
|---|---|---|
| 核心对象 | 商品 ID / SKU | 商品规格 + 有效单位 |
| 比较方式 | 同款比价 | 规格归一化比价 |
| 例子 | 可口可乐 330ml×24 多少钱 | 每 100ml 多少钱、每瓶多少钱、每升多少钱 |
| 难点 | 商品匹配 | 规格理解、单位换算、组合拆解 |
| 用户价值 | 哪个平台便宜 | 哪个规格真正划算 |
| 适合场景 | 电商横向比价 | 会员店、商超、批发、家庭囤货 |

你这个方向更“细”，也更容易做出差异化。

普通比价系统看到：

> 可口可乐 330ml×24，40 元  
> 可口可乐 500ml×12，38 元

可能只会说：

> 第一个每瓶 1.67 元，第二个每瓶 3.17 元。

但你的系统应该说：

| 商品 | 价格 | 包装规格 | 每瓶 | 每 100ml | 每升 |
|---|---:|---:|---:|---:|---:|
| 330ml × 24 | 40 元 | 7920ml | 1.67 元 | 0.51 元 | 5.05 元 |
| 500ml × 12 | 38 元 | 6000ml | 3.17 元 | 0.63 元 | 6.33 元 |

然后得出：

> 虽然 500ml 那款“每瓶更贵”，但关键不是瓶价，而是容量单价。330ml×24 实际每 100ml 更便宜约 19%。

这才是你的产品价值。

---

# 二、系统核心不是“比价”，而是“规格语义解析”

你真正要解决的是这类问题：

> “1L×6瓶”  
> “330ml×24听”  
> “250ml*12盒*2提”  
> “家庭装 1.25L*6”  
> “买一送一”  
> “第二件半价”  
> “组合装 3瓶洗发水 500ml+护发素 500ml”  
> “净含量 1.8kg，内含 30 小包”  
> “每包约 60g”  
> “冷冻鸡胸肉 2kg，约 8-10片”  
> “抽纸 4层 100抽 24包”

所以这个系统的核心能力应该是：

## **从商品标题、图片、详情、规格参数中提取结构化规格信息**

例如：

```json
{
  "商品名": "可口可乐 汽水 330ml*24听",
  "品牌": "可口可乐",
  "品类": "碳酸饮料",
  "包装形式": "听装",
  "单件容量": {
    "value": 330,
    "unit": "ml"
  },
  "数量": 24,
  "总容量": {
    "value": 7920,
    "unit": "ml"
  },
  "价格": 40,
  "可比较单位": [
    "per_100ml",
    "per_liter",
    "per_can"
  ],
  "每100ml价格": 0.505,
  "每升价格": 5.05,
  "每听价格": 1.67
}
```

这比单纯爬价格复杂得多，但也正因为复杂，普通比价插件不容易做好。

---

# 三、建议的系统架构

我建议你不要一开始就做“全品类通用比价”，而是做一个**插件式的规格解析 + 单价归一化引擎**。

整体可以拆成 5 层：

```text
数据采集层
   ↓
商品规格解析层
   ↓
规格归一化层
   ↓
单位价格计算层
   ↓
比价/推荐/展示层
```

---

## 1. 数据采集层

先期可以只接：

- 山姆会员店
- Costco 类商城
- 盒马 / 麦德龙
- 京东自营大包装商品
- 天猫超市
- 用户手动录入 / 截图识别

数据来源可以有几种：

| 来源 | 难度 | 适合阶段 |
|---|---:|---|
| 手动复制商品标题和价格 | 低 | MVP |
| 商品详情页 URL 解析 | 中 | 早期插件 |
| 浏览器插件读取页面 DOM | 中 | 插件化方向 |
| 手机截图 OCR | 中高 | 后期增强 |
| 小程序/APP 抓包 | 高，且有合规风险 | 不建议初期做 |
| 官方 API | 低到中，但少见 | 有资源时做 |

**MVP 阶段建议：**

先做一个浏览器插件或 Web 页面，让用户输入：

```text
可口可乐 汽水 330ml*24听
价格：40元
```

系统自动输出：

```text
每听：1.67元
每100ml：0.51元
每升：5.05元
```

先别急着爬全站。你现在最需要验证的是：**规格解析准确率和单位价格体验是否有价值**。

---

## 2. 商品规格解析层

这是最关键的部分。

可以采用：

```text
规则引擎 + AI 解析 + 置信度校验
```

不要完全依赖 AI，也不要完全依赖正则。

### 为什么不能只用正则？

因为商品标题非常脏：

```text
农夫山泉 饮用天然水 550ml*24瓶 整箱
可口可乐 零度 迷你罐 200ml×12罐
巴黎水气泡水 原味 330ml*24瓶/箱
维他柠檬茶 250ml*24盒
伊利安慕希 205g*12盒
清风抽纸 原木纯品 3层100抽*24包
奥妙洗衣凝珠 100颗
蓝月亮洗衣液 3kg+500g补充装
```

有些是：

- `ml`
- `mL`
- `毫升`
- `L`
- `升`
- `g`
- `kg`
- `克`
- `斤`
- `磅`
- `袋`
- `盒`
- `瓶`
- `罐`
- `听`
- `包`
- `片`
- `抽`
- `卷`

还有组合装：

```text
500ml*3瓶 + 300ml*2瓶
1.5L*6瓶 + 330ml*6罐
2kg装，内含10小包
买2件减20
```

正则能处理 70%，剩下 30% 会很麻烦。

### 为什么不能只用 AI？

因为 AI 有时会“自信地算错”。

例如：

```text
250ml*24盒*2箱
```

AI 可能只算成：

```text
250ml * 24 = 6000ml
```

而漏掉 `*2箱`。

所以最稳的方案是：

## **规则先解析，AI 补充理解，最后计算器确定结果**

可以这样设计：

```text
原始标题
  ↓
正则提取候选规格
  ↓
AI 结构化解析
  ↓
规则校验
  ↓
统一单位
  ↓
价格计算
```

# 四、插件式设计应该怎么做

你说想做成“插件式比价工具”，这个思路是对的。

我建议拆成两类插件：

## 1. 商店插件 Store Plugin

负责从不同商店获取商品信息。

例如：

```ts
interface StorePlugin {
  id: string;
  name: string;

  matchUrl(url: string): boolean;

  extractProduct(page: Document): Promise<RawProduct>;
}
```

`RawProduct` 结构：

```ts
interface RawProduct {
  store: string;
  url?: string;
  title: string;
  subtitle?: string;
  price: number;
  originalPrice?: number;
  promotionText?: string;
  imageUrls?: string[];
  specText?: string;
  detailText?: string;
  skuId?: string;
}
```

比如山姆插件只负责：

```json
{
  "store": "sam",
  "title": "可口可乐汽水 330ml*24听",
  "price": 40,
  "promotionText": "会员价"
}
```

它不负责算单价。

---

## 2. 品类解析插件 Category Plugin

负责不同品类的单位归一化。

例如：

```ts
interface CategoryPlugin {
  category: string;

  detect(product: RawProduct): boolean;

  parseSpec(product: RawProduct): Promise<ParsedSpec>;

  getComparableUnits(spec: ParsedSpec): ComparableUnit[];
}
```

饮料类插件：

```ts
category = "beverage"

comparableUnits = [
  "per_100ml",
  "per_liter",
  "per_bottle",
  "per_can"
]
```

纸品类插件：

```ts
category = "paper"

comparableUnits = [
  "per_100_sheets",
  "per_pack",
  "per_roll",
  "per_sheet"
]
```

洗衣液类插件：

```ts
category = "laundry_liquid"

comparableUnits = [
  "per_100g",
  "per_kg",
  "per_load"
]
```

鸡胸肉/牛肉类插件：

```ts
category = "meat"

comparableUnits = [
  "per_100g",
  "per_kg",
  "per_piece"
]
```

这样以后扩展品类会比较自然。

# 五、核心数据模型建议

你需要一个非常明确的内部结构。

## RawProduct：原始商品

```json
{
  "id": "sam_123456",
  "store": "sam",
  "title": "可口可乐汽水 330ml*24听",
  "price": 40,
  "currency": "CNY",
  "promotionText": "",
  "sourceUrl": "https://..."
}
```

---

## ParsedSpec：解析后的规格

```json
{
  "category": "beverage",
  "brand": "可口可乐",
  "productName": "可口可乐汽水",
  "packageType": "can",
  "unitSize": {
    "value": 330,
    "unit": "ml"
  },
  "quantity": 24,
  "multipliers": [24],
  "totalAmount": {
    "value": 7920,
    "unit": "ml"
  },
  "confidence": 0.96,
  "evidence": [
    "标题中识别到 330ml*24听"
  ]
}
```

---

## UnitPrice：计算后的单位价格

```json
{
  "price": 40,
  "normalized": {
    "per_100ml": 0.505,
    "per_liter": 5.05,
    "per_can": 1.667
  },
  "bestDisplayUnit": "per_100ml",
  "warnings": []
}
```

---

## ComparisonItem：用于比较的对象

```json
{
  "productId": "sam_123456",
  "displayName": "可口可乐 330ml*24听",
  "store": "山姆",
  "price": 40,
  "totalAmount": "7.92L",
  "unitPrice": {
    "per_100ml": 0.51,
    "per_liter": 5.05,
    "per_can": 1.67
  },
  "score": 92
}
```

# 六、AI 在系统里应该扮演什么角色？

这里建议你把 AI 当成：

## **规格理解器，不是价格计算器**

AI 负责把脏文本转成结构化信息。

计算必须由程序完成。

### 不推荐

```text
直接问 AI：
这个商品每100ml多少钱？
```

因为它可能算错、漏算、四舍五入混乱。

---

### 推荐

第一步，让 AI 提取规格：

```json
{
  "title": "可口可乐 汽水 330ml*24听",
  "price": 40
}
```

让 AI 输出：

```json
{
  "category": "beverage",
  "unit_size_value": 330,
  "unit_size_unit": "ml",
  "quantity": 24,
  "package_unit": "can",
  "total_amount_value": 7920,
  "total_amount_unit": "ml",
  "confidence": 0.96
}
```

第二步，程序计算：

```ts
per100ml = price / totalMl * 100
perLiter = price / totalMl * 1000
perPiece = price / quantity
```

第三步，程序校验：

```ts
if (totalMl !== unitSize * quantity) {
  warning.push("规格计算不一致，需要人工确认")
}
```

这样可靠性高很多。

# 七、AI 解析 Prompt 可以这样设计

你可以让 AI 严格输出 JSON Schema。

例如：

```text
你是商品规格解析器。请从商品标题、规格参数、促销文本中提取结构化规格信息。

要求：
1. 只输出 JSON。
2. 不要计算价格。
3. 如果无法确定，请填 null。
4. 需要给出 confidence。
5. total_amount 只在规格明确时给出。
6. 不要猜测标题中没有的信息。

输入：
商品标题：可口可乐 汽水 330ml*24听
价格：40元
规格参数：无

输出格式：
{
  "category": "",
  "brand": "",
  "product_name": "",
  "unit_size": {
    "value": null,
    "unit": ""
  },
  "quantity": null,
  "package_unit": "",
  "multipliers": [],
  "total_amount": {
    "value": null,
    "unit": ""
  },
  "confidence": 0,
  "evidence": []
}
```

更工程化一点，可以直接上 function calling / JSON schema。

# 八、不要只做饮料，品类要分阶段

你的 MVP 最适合从**饮料**开始，因为饮料的规格相对清晰：

```text
ml / L × 瓶数 / 罐数 / 盒数
```

## 第一阶段：饮料

包括：

- 碳酸饮料
- 果汁
- 咖啡
- 茶饮料
- 牛奶
- 酸奶
- 矿泉水
- 气泡水

核心单位：

| 口径 | 说明 |
|---|---|
| 每瓶/每罐 | 用户直觉强 |
| 每 100ml | 真实可比 |
| 每升 | 大规格饮料更直观 |

---

## 第二阶段：食品

比如：

- 坚果
- 鸡胸肉
- 牛肉
- 零食
- 冷冻食品
- 方便食品

核心单位：

| 口径 | 说明 |
|---|---|
| 每 100g | 最通用 |
| 每 kg | 肉类、冷冻食品 |
| 每包/每袋 | 便携食品 |

---

## 第三阶段：日用品

比如：

- 洗衣液
- 洗衣凝珠
- 抽纸
- 卷纸
- 湿巾
- 洗发水
- 沐浴露

这类会更复杂。

比如抽纸：

```text
4层 100抽 24包
```

到底比较：

- 每包多少钱？
- 每抽多少钱？
- 每 100 抽多少钱？
- 是否要考虑层数？
- 4 层 100 抽和 3 层 120 抽是否直接可比？

这里就进入“质量修正”的问题了，暂时不要一开始做。

# 十、关于“部分商品不参与比价”的处理

这点很重要。

你的系统不应该强行把所有商品都拉进同一个榜单。

应该有一个字段：

```json
{
  "comparable": true,
  "comparisonGroup": "cola_regular",
  "excludedReason": null
}
```

或者：

```json
{
  "comparable": false,
  "excludedReason": "规格不完整，无法确认总容量"
}
```

常见不可比原因：

| 原因 | 例子 |
|---|---|
| 规格缺失 | “超值家庭装”但没写容量 |
| 组合混杂 | “饮料零食礼包” |
| 赠品复杂 | “买饮料送杯子” |
| 口味差异明显 | 普通可乐 vs 无糖可乐是否同组，看策略 |
| 品质等级不同 | 原切牛排 vs 调理牛排 |
| 单位不统一 | 抽纸按抽数，湿巾按片数 |
| 促销条件复杂 | 满 399 减 80，需要凑单 |

这样你就不会把系统做成“万物皆可比”的玄学排行榜。

# 十一、比价不只是最低价，还应该有解释

这个产品如果只显示：

> A 每 100ml 0.51 元  
> B 每 100ml 0.63 元

还不够。

最好给用户一句自然语言解释：

```text
A 看起来每瓶便宜，实际也更便宜。折算后每 100ml 约 0.51 元，比 B 低约 19%。
```

或者：

```text
B 每瓶价格更高，但单瓶容量更大。折算后每 100ml 反而比 A 便宜约 8%。
```

这就是 AI 可以参与的第二个地方：

## **把计算结果解释成人话**

但注意：解释可以用 AI，计算不要交给 AI。

# 十二、MVP 可以怎么做？

我建议你的第一个版本非常克制：

## MVP 目标

只支持：

- 饮料类
- 手动输入商品标题 + 价格
- 支持 `ml/L × 数量`
- 输出：
  - 每瓶/罐/盒价格
  - 每 100ml 价格
  - 每升价格
  - 规格解析置信度
  - 是否适合参与比价

---

## MVP 页面示例

用户输入：

```text
可口可乐 零度汽水 330ml*24听
价格：40
```

系统输出：

```text
识别结果：
- 品类：饮料
- 单罐容量：330ml
- 数量：24听
- 总容量：7.92L

价格：
- 每听：1.67元
- 每100ml：0.51元
- 每升：5.05元

结论：
这款适合按每100ml参与饮料比价。
```

然后用户再输入：

```text
百事可乐 无糖 500ml*12瓶
价格：38
```

系统比较：

| 商品 | 价格 | 总容量 | 每瓶 | 每 100ml | 每升 |
|---|---:|---:|---:|---:|---:|
| 可口可乐 330ml×24 | 40 | 7.92L | 1.67 | 0.51 | 5.05 |
| 百事可乐 500ml×12 | 38 | 6L | 3.17 | 0.63 | 6.33 |

结论：

```text
可口可乐 330ml×24 的真实容量单价更低，每100ml约便宜 19.7%。
```

# 十三、推荐的技术实现

以你的背景，推荐这样做。

## 后端

可以用：

- Python + FastAPI

# 十四、规格解析建议采用“三段式”

## 第一段：规则解析

先用正则抽取：

```regex
(\d+(\.\d+)?)\s*(ml|mL|毫升|L|升|g|kg|克|千克)\s*[*xX×]\s*(\d+)
```

处理：

```text
330ml*24
1L×6
250ml x 12
500g*3袋
```

---

## 第二段：AI 补充解析

处理复杂文本：

```text
250ml*12盒*2箱
500ml*3瓶+300ml*2瓶
1.8kg，内含30小包
每盒12瓶，每瓶200ml，共2盒
```

---

## 第三段：规则校验和单位计算

程序统一换算：

```text
1L = 1000ml
1kg = 1000g
1斤 = 500g
```

计算：

```ts
totalAmount = unitSize * quantity * multiplier
unitPrice = price / totalAmount
```

所有计算都留痕：

```json
{
  "formula": "40 / (330 * 24) * 100",
  "result": 0.505
}
```

这个留痕很重要，将来用户质疑时可以解释。

# 十五、复杂促销怎么处理？

促销是另一个坑。

先分三类：

## 1. 直接价

```text
会员价 40元
```

最简单。

---

## 2. 件数促销

```text
第二件半价
买一送一
2件8折
```

需要计算有效价格。

例如：

```text
单价 40 元，第二件半价
```

如果买 2 件：

```text
有效总价 = 40 + 20 = 60
单件有效价 = 30
```

---

## 3. 满减促销

```text
满 299 减 50
满 399 减 80
```

这个不要强行算成单品价格。

应该展示为：

```text
当前价格：40元
满减后理论最低价：约33.3元
条件：需要参与满399减80，且凑单比例完全均摊
可信度：中
```

满减均摊很容易误导用户。

所以你可以设计两个价格：

```json
{
  "basePrice": 40,
  "effectivePrice": 33.3,
  "effectivePriceType": "promotion_allocated",
  "confidence": 0.65
}
```

---

# 十六、真正有差异化的功能：反套路提示

你这个系统最有意思的地方不是算数，而是帮用户识别“看起来便宜但其实不便宜”。

比如：

## 1. 每瓶便宜陷阱

```text
A：330ml×24，40元，每瓶1.67
B：500ml×12，38元，每瓶3.17
```

系统提示：

> B 每瓶更贵，但容量也更大。请优先看每 100ml，而不是每瓶价格。

---

## 2. 大包装未必便宜

```text
A：1L×6，48元
B：330ml×24，40元
```

系统提示：

> A 是大瓶装，但每升价格约 8 元；B 每升约 5.05 元。大包装并不一定更划算。

---

## 3. 组合装混淆

```text
洗发水 750ml + 护发素 500ml，99元
```

系统提示：

> 该商品包含不同用途商品，不适合直接按总 ml 与单品洗发水比较。

这就是产品体验上的护城河。

---

# 十七、商品匹配策略

你不一定要一开始就做“同款匹配”。

先做三个层级：

## Level 1：同标题/同 SKU

最简单。

```text
可口可乐 330ml×24
```

---

## Level 2：同品牌同品类

```text
可口可乐 普通
可口可乐 零度
可口可乐 纤维+
```

可以分组但不强行判定完全同类。

---

## Level 3：同消费场景

```text
无糖可乐：
- 可口可乐零度
- 百事无糖
- 元气森林可乐味气泡水
```

这个才需要 AI/Embedding。

但 MVP 不要一开始就做 Level 3，否则会陷入“百事和可口是不是同类”的哲学战争。价格系统里最怕的不是 bug，是用户拿元气森林来碰瓷可乐。

# 十九、AI 解析结果一定要有置信度和证据

这点非常重要。

例如：

```json
{
  "confidence": 0.92,
  "evidence": [
    "标题中出现 330ml*24听",
    "听 被识别为 can",
    "饮料类默认主比较单位为 per_100ml"
  ]
}
```

如果置信度低：

```json
{
  "confidence": 0.48,
  "warnings": [
    "标题中出现家庭装，但未识别到明确数量",
    "详情文本缺失"
  ]
}
```

前端可以显示：

```text
规格识别可信度：较低，建议人工确认。
```

这样用户会更信任系统。

# 二十、人工纠错机制很关键

早期一定要允许用户修正：

```text
系统识别：
330ml × 12

用户修正：
330ml × 24
```

然后保存为：

```json
{
  "parse_source": "manual_corrected"
}
```

未来可以用这些数据微调规则或做 few-shot 示例。

你这个系统不是一次性做准，而是越用越准。

# 二十一、插件式系统的最终形态

理想形态可以是：

```text
浏览器插件 / 小程序 / App
   ↓
识别当前商品
   ↓
解析规格
   ↓
计算真实单位价格
   ↓
对比同类商品
   ↓
给出“是否值得买”结论
```

例如用户在山姆商品页看到：

```text
农夫山泉 550ml*24瓶
价格：39.9
```

插件浮层：

```text
真实单价：
- 每瓶：1.66元
- 每100ml：0.30元
- 每升：3.02元

在已收录矿泉水中：
- 低于同类均价 12%
- 当前属于较划算
```

再比如：

```text
某品牌气泡水 480ml*15瓶
价格：69.9
```

插件提示：

```text
每100ml：0.97元
比当前可比较气泡水均价高约 26%。

注意：该商品每瓶看似不到5元，但容量只有480ml。
```

这个就很有杀伤力。

# 二十三、我会推荐的核心产品原则

## 1. 不追求万物可比

要明确告诉用户：

> 这个商品不适合直接参与比价。

这是专业感，不是缺陷。

---

## 2. 默认用最稳定的单位

饮料默认：

```text
每100ml
```

食品默认：

```text
每100g
```

大宗食品默认：

```text
每kg
```

纸品默认：

```text
每100抽 / 每卷
```

---

## 3. AI 只做理解，不做最终判断

最终价格、单位换算、是否可比，尽量由规则引擎控制。

---

## 4. 每个结论都能解释

比如：

```text
计算方式：40 ÷ 24 ÷ 330 × 100 = 0.51元/100ml
```

这会让用户觉得靠谱。

---

## 5. 不要一开始做大而全

第一版只做：

```text
饮料 + 山姆 + 每100ml
```

这个范围已经足够验证价值。

# 二十五、最终建议

你这个项目的关键不是“接 AI”，而是：

## **用 AI 把非结构化商品规格变成结构化数据，再用确定性程序做单位价格计算。**

我建议你把系统核心定义为：

```text
商品规格解析器
+
单位价格归一化引擎
+
可比较性判断器
+
插件式商店适配器
```

最小可行版本：

```text
山姆/类山姆商品
饮料品类
标题 + 价格解析
每100ml / 每升 / 每瓶计算
可比性判断
人工纠错
```

先把这个做准，就已经很有用了。

尤其山姆这种会员店，经常出现：

> “看起来一瓶 3 块多，很便宜”  
> 但实际上容量、瓶数、促销、组合一换算，可能完全不是那么回事。

你的系统要做的，就是把这些“超市数学障眼法”拆开。  
说白了，这是给消费者装一个**价格透视眼**。
